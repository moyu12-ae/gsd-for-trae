# GSD Debugger - 调试技能

## 概述

GSD Debugger 负责使用科学方法调查和诊断问题：收集症状、形成假设、测试假设、找到根因、提供修复方案。

## 核心职责

1. **证据收集** - 收集症状、错误日志、相关代码
2. **假设形成** - 生成多个假设，按可能性排序
3. **假设测试** - 设计实验验证/反驳假设
4. **问题修复** - 实现最小修复，添加回归测试
5. **根因调查** - 必先于任何修复
6. **架构讨论** - 3 次修复失败后触发

## 四阶段系统化调试流程

**铁律**: 在尝试任何修复之前，必须先完成根因调查。

```
不调查根因 = 必然失败
```

### Phase 1: 根因调查 (必先于修复)

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完全阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现问题**
   - 能可靠地触发吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么更改可能导致这个问题？
   - Git diff，最近的提交
   - 新依赖，配置更改
   - 环境差异

4. **多组件系统中的证据收集**

   **当系统有多个组件时（CI → build → 签名，API → 服务 → 数据库）：**
   
   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每一层的状态
   
   运行一次以收集证据显示哪里出了问题
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时：**
   - 坏值从哪里来？
   - 什么用坏值调用了这个？
   - 一直追踪到找到源头
   - 在源头修复，而不是症状

### Phase 2: 模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中找到类似的正常工作代码
   - 什么与坏的一样工作？

2. **与参考对比**
   - 如果正在实现模式，完全阅读参考实现
   - 不要略读 - 每一行都读
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作和坏的区别是什么？
   - 列出每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 这个还需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### Phase 3: 假设和测试

**科学方法：**

1. **形成单一假设**
   - 清楚地陈述："我认为 X 是根本原因因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小可能的更改来测试假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **继续前验证**
   - 成功了吗？是 → Phase 4
   - 没成功？形成新假设
   - 不要在上面添加更多修复

4. **当你不明白时**
   - 说 "我不明白 X"
   - 不要假装明白
   - 寻求帮助
   - 做更多研究

### Phase 4: 实现

**修复根本原因，而不是症状：**

1. **创建失败的测试用例**
   - 最简单的可能复现
   - 如果可能自动化测试
   - 如果没有框架则一次性测试脚本
   - 修复前必须有
   - 使用 `superpowers:test-driven-development` 技能编写适当的失败测试

2. **实现单一修复**
   - 解决已识别的根本原因
   - 一次更改一个
   - 不要"既然在这里"的改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没坏吧？
   - 问题真的解决了吗？

4. **如果修复不工作**
   - 停止
   - 数数：你尝试了多少次修复？
   - 如果 < 3：返回 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下文）**
   - 不要在没有架构讨论的情况下尝试修复 #4

### Phase 4.5: 如果 3+ 修复失败 - 质疑架构

**表明架构问题的模式：**
- 每个修复在不同地方揭示新的共享状态/耦合/问题
- 修复需要"大规模重构"来实现
- 每个修复在其他地方创建新症状

**停止并质疑基础：**
- 这个模式根本健全吗？
- 我们是否"纯粹因为惯性而坚持"？
- 我们应该重构架构而不是继续修复症状吗？

**在尝试更多修复之前与人类讨论**

这不是失败的假设 - 这是错误的架构。

## 调试红牌 - 停止并遵循流程

如果发现自己想：
- "先快速修复，以后再调查"
- "只是尝试改 X 看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "我看到问题了，让我修复它"
- "我不完全明白但这可能有效"
- "模式说 X 但我会 differently adapt 它"
- "以下是主要问题：[列出没有调查的修复]"
- 在追踪数据流之前提出解决方案
- **"再尝试一次修复"（已经尝试了 2+ 次）**
- **每个修复在不同地方揭示新问题**

**所有这些意味着：停止。返回 Phase 1。**

**如果 3+ 修复失败：** 质疑架构（见 Phase 4.5）

## 工作流程

### 调试前置流程：自动上下文感知

在启动正式调试流程前，系统将自动执行以下上下文获取步骤：

#### 步骤1: 读取GSD系统上下文文件

- 自动查找项目根目录下 `.planning/` 目录中的所有规划文件
- **优先读取文件：**
  - `.planning/STATE.md` - 当前项目状态、阶段、任务、进度
  - `.planning/CONTEXT.md` - 项目描述、用户决策、约束条件
  - `.planning/PLAN.md` - 当前执行计划和任务
  - `.planning/SUMMARY.md` - 执行总结
  - `.planning/research/STACK.md` - 技术栈
  - `.planning/research/ARCHITECTURE.md` - 项目架构
  - `.planning/research/FEATURES.md` - 已完成/待开发功能
- 解析项目当前进度、已完成功能、待开发任务和技术决策

#### 步骤2: 查找现有DEBUG.md文档

- 在 `.planning/` 目录查找已有的 `DEBUG.md` 文件
- 如果存在，自动读取文件内容，了解：
  - 当前调试进度和状态
  - 已经尝试过的调试方案及其结果
  - 已被标记为失败的方案（将自动规避）
  - 已收集的证据和已验证的假设

#### 步骤3: 初始化调试上下文

- 新项目：在 `.planning/` 目录创建 `DEBUG.md` 模板，写入项目上下文信息
- 已有项目：基于现有 `.planning/DEBUG.md` 继续调试流程，跳过已失败的方案

### 多轮调试支持

本技能支持多轮持续调试，每轮调试自动管理上下文和历史记录。

#### 轮次管理规则

| 用户表达                 | 行为             |
| -------------------- | -------------- |
| "重新debug"、"新一轮debug" | 创建新轮次，上一轮自动结束  |
| "继续调试"、"接着调"         | 继续当前轮次         |
| 修复完成或用户确认            | 当前轮次结束，汇总到长期文档 |

#### 轮次文件结构

```
.planning/
├── DEBUG.md                    # 长期调试文档（所有轮次汇总）
├── DEBUG_ROUND_1.md           # 第1轮调试详情
├── DEBUG_ROUND_2.md           # 第2轮调试详情
└── DEBUG_ROUND_N.md           # 第N轮调试详情
```

#### 轮次检测逻辑

**新轮次识别：**

1. 检查 `.planning/DEBUG.md` 中的 `current_round` 字段
2. 如果用户说"重新debug"、"新一轮debug"，当前轮次结束
3. 创建新的 `DEBUG_ROUND_{N+1}.md` 文件

**新轮次工作流程：**

```
新轮次开始
    ↓
读取 .planning/DEBUG.md (长期文档)
    ↓
读取 .planning/DEBUG_ROUND_{N}.md (上一轮详情)
    ↓
分析历史调试方案和失败记录
    ↓
制定新轮次调试计划（避免重复失败方案）
    ↓
执行调试
    ↓
轮次结束 → 汇总到 DEBUG.md
```

#### 轮次结束处理

当轮次结束时（用户确认或修复完成）：

1. 将本轮调试详情复制到 `DEBUG_ROUND_N.md`
2. 提取关键信息汇总到 `DEBUG.md`：
   - 问题描述
   - 根因分析
   - 修复方案
   - 验证结果
3. 更新 `DEBUG.md` 中的 `current_round` 和 `last_updated` 字段

#### DEBUG.md 长期文档格式

```yaml
---
current_round: 3
last_round_ended: "2024-01-15T10:00:00Z"
total_rounds: 3
---

# Debug History

## Round 1 (2024-01-10)
- 问题: 用户无法登录
- 根因: 数据库连接超时
- 状态: ✓ 已解决

## Round 2 (2024-01-12)
- 问题: 支付失败
- 根因: API密钥过期
- 状态: ✓ 已解决

## Round 3 (2024-01-15)
- 问题: [当前问题]
- 状态: 进行中
```

#### DEBUG\_ROUND\_N.md 单轮格式

```yaml
---
round: 2
started_at: "2024-01-12T09:00:00Z"
ended_at: "2024-01-12T11:30:00Z"
issue: "支付失败"
status: resolved|failed|in_progress
previous_round_ref: 1
---

# Round 2: 支付失败

## 问题描述
[用户描述的具体问题]

## 历史参考
- 上一轮尝试: 数据库连接问题（已解决）
- 避免重复: 不使用相同的API端点

## 调试过程
[本轮详细的调试步骤和结果]

## 结论
- 根因: API密钥过期
- 修复: 更新密钥
- 状态: 已验证
```

### 调试过程持久化机制

每一轮调试测试完成后，无论成功或失败，系统将自动更新DEBUG.md文档：

#### 成功场景更新

当调试方案测试成功后：

1. 记录根因分析结果
2. 记录完整修复方案和代码变更
3. 记录验证步骤和结果
4. 添加预防措施建议
5. 标记调试状态为RESOLVED

#### 失败场景更新

当调试方案测试失败后：

1. 标记该假设为DISPROVED
2. 详细记录失败原因和观察到的现象
3. 记录执行的所有操作和输出结果
4. 该方案将被自动加入失败列表，后续调试不再尝试相同方案

#### 自动更新规则

- 每次假设测试后立即更新，无需等待整个调试完成
- 保留完整的调试历史，便于回溯和经验积累
- 所有更新自动完成，无需手动干预

## 核心调试流程（四阶段方法论）

本调试流程整合了系统化调试方法论，遵循四个核心阶段，每个阶段必须完成后才能进入下一阶段：

### 🚫 铁律：未找到根因前，禁止提出任何修复方案

**违反此流程就是违反调试精神，随机修复只会浪费时间并引入新bug。**

***

## 阶段 0: 症状确认（使用 AskUserQuestion）

**⚠️ 在开始调试前，确认你对问题的理解**

使用 AskUserQuestion 确认问题描述：

```markdown
AskUserQuestion(
  questions: [
    {
      header: "症状",
      question: "你描述的问题是：我理解[复述用户的问题描述]，对吗？",
      options: [
        {
          label: "是的，正确",
          description: "你的理解完全正确，继续调试"
        },
        {
          label: "部分偏差",
          description: "主要问题对，但有细节需要补充"
        },
        {
          label: "完全不对",
          description: "让我重新描述问题"
        }
      ],
      multiSelect: false
    }
  ]
)
```

**根据用户选择：**

- "是的，正确"：继续调试
- "部分偏差"：请用户补充细节
- "完全不对"：让用户重新描述问题

***

## 阶段 1: 根因调查 (Root Cause Investigation)

**在尝试任何修复前，必须完成此阶段：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告信息，它们通常包含直接解决方案
   - 完整阅读堆栈跟踪，记录行号、文件路径、错误代码
2. **稳定复现问题**
   - 能否可靠触发问题？确切步骤是什么？
   - 是否每次都发生？如果无法复现 → 收集更多数据，不要猜测
3. **检查最近变更**
   - 什么变更可能导致这个问题？Git diff、最近提交
   - 新依赖、配置变更、环境差异
4. **多组件系统收集证据**
   **当系统包含多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次收集证据，显示哪里出问题
   然后分析证据识别失败组件
   再调查该特定组件
   ```
5. **跟踪数据流**
   - 错误值来自哪里？谁调用了这个函数并传入了错误值？
   - 持续向上追溯直到找到源头，在源头修复，而不是在症状处修复
   - 参考 [root-cause-tracing.md](./references/root-cause-tracing.md) 完整回溯技术

***

## 阶段 2: 模式分析 (Pattern Analysis)

**在修复前找到模式：**

1. **找到工作示例**
   - 在同一代码库中定位类似的可工作代码
   - 什么与损坏部分类似但能正常工作？
2. **对比参考实现**
   - 如果实现的是已知模式，完整阅读参考实现
   - 不要略读，理解完整模式后再应用
3. **识别差异**
   - 工作和损坏的代码之间有什么不同？列出所有差异，无论多小
   - 不要假设"这不重要"
4. **理解依赖**
   - 这个组件需要哪些其他组件？什么设置、配置、环境？
   - 它做了哪些假设？

***

## 阶段 3: 假设和测试 (Hypothesis and Testing)

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为X是根本原因，因为Y"
   - 写下来，具体明确，不要模糊
2. **最小化测试**
   - 做最小可能的变更来测试假设
   - 一次只改变一个变量，不要同时修复多个问题
3. **验证后再继续**
   - 生效？是 → 进入阶段4
   - 不生效？形成新假设，，不要在上面叠加更多修复
4. **当你不知道时**
   - 说"我不理解X"，不要假装知道
   - 请求帮助，继续研究
5. **三次失败后质疑架构**
   - 如果尝试了3次以上修复都失败，停止并质疑架构是否存在根本问题
   - 每个修复都在不同地方暴露出新的共享状态/耦合/问题 → 架构问题

***

## 阶段 4: 实现修复 (Implementation)

**修复根因，而不是症状：**

1. **创建失败测试用例**
   - 最简单的复现代码，自动化测试如果可能
   - 修复前必须有失败的测试用例
2. **实现单一修复**
   - 解决已识别的根因，一次只做一个变更
   - 没有"顺便"的改进，没有捆绑重构
3. **验证修复**
   - 测试现在通过了吗？其他测试没有被破坏吗？
   - 问题真的解决了吗？
4. **添加深度防御验证**
   - 在数据经过的每一层添加验证，使bug从结构上不可能发生
   - 参考 [defense-in-depth.md](./references/defense-in-depth.md) 多层验证模式
   - 使用条件等待替代任意超时，减少测试不稳定性，参考 [condition-based-waiting.md](./references/condition-based-waiting.md)

***

## 原有工作流映射

现有调试步骤已整合到上述四阶段流程中：

- 原Step 1（收集证据）→ 阶段1（根因调查）
- 原Step 2（形成假设）→ 阶段2（模式分析）输出
- 原Step 3（测试假设）→ 阶段3（假设和测试）
- 原Step 4（找到根因）→ 阶段3完成标志
- 原Step 5（提供修复方案）→ 阶段4（实现修复）
- 原Step 6（生成DEBUG.md）→ 全流程输出

### 假设生成模板

```yaml
hypotheses:
  - id: H1
```

## 调试原则

1. **科学方法** - 假设→测试→结论循环
2. **多个假设** - 不要只找一个答案
3. **证据驱动** - 所有结论都要有证据支持
4. **最小修复** - 修复根因，不要过度工程

## 最佳实践

1. **复现问题** - 先复现再调试
2. **版本控制** - 检查最近的变更
3. **二分查找** - 使用 git bisect 快速定位
4. **记录过程** - 完整记录调试过程
5. **添加测试** - 修复后添加回归测试

## 与GSD系统其他模块的交互

### 与 gsd-context 交互

- **交互方式**: 直接读写 `.planning/` 目录下的文件
- **读取**: 使用 `Read` 工具读取 `.planning/STATE.md`、`.planning/CONTEXT.md` 等文件
- **写入**: 使用 `Write` 或 `SearchReplace` 工具更新状态
- **调用 gsd-context**: 通过 `Task` 工具调用 `gsd-context` 的操作（如保存会话快照）

### 与 gsd-executor 交互

- **调用时机**: 根因确认并设计好修复方案后
- **交互方式**: 通过 `gsd-orchestrator` 调度，或直接使用 `Task` 工具调用
- **输出**: 将修复方案写入 `.planning/PLAN.md`，供 executor 执行

### 与 gsd-verifier 交互

- **调用时机**: 修复方案执行完成后
- **交互方式**: 通过 `gsd-orchestrator` 调度，或直接使用 `Task` 工具调用
- **触发验证**: 验证修复方案是否有效

### 与 gsd-orchestrator 交互

- **被调用**: 当用户说"我遇到问题了"、"调试"等关键词时，由 orchestrator 调度
- **返回结果**: 将调试结果写入 `.planning/DEBUG.md`，更新 `.planning/STATE.md`

## 与其他 Skills 的关系

- **输入**: 用户报告的问题或 gsd-verifier 发现的差距
- **输出**: `.planning/DEBUG.md` 报告和修复方案
- **状态**: 通过 gsd-context 管理 `.planning/` 目录下的文件
- **后续**: 修复后由 gsd-orchestrator 决定下一步（执行、验证等）

## 调试完成后保存会话快照 (必须执行)

**⚠️ 重要：调试完成后必须保存会话状态**

在调试完成后（阶段 4 实现修复完成后），使用 Write 工具将以下内容追加到 `.planning/SESSION.md`：

```markdown

**Problem:** [问题描述]

**Solution:** [修复方案]

```

**注意**：如果需要执行修复，还需要在 SESSION.md 中记录调用 gsd-executor 的信息。
