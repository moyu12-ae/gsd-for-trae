# GSD 规划完整性检查清单

## 概述

此清单用于在规划前检查项目是否具备足够的规划信息，确保生成的计划全面且可执行。

## 检查项目

### 1. 功能需求 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 核心功能列表完整 | □ | 列出所有必须实现的功能 |
| 功能优先级已定义 | □ | Must Have / Should Have / Could Have / Won't Have |
| 用户场景已描述 | □ | 描述每个功能的典型使用场景 |
| 验收标准已明确 | □ | 每个功能有可测试的验收条件 |

### 2. 非功能需求 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 性能要求已明确 | □ | 响应时间、并发量、数据处理量 |
| 安全要求已考虑 | □ | 认证、授权、数据加密、API 保护 |
| 可用性要求 | □ | 兼容性、错误恢复、降级策略 |
| 可扩展性 | □ | 未来扩展方向和技术预留 |

### 3. 技术方案 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 数据模型已设计 | □ | 关键数据对象、关系、ER 图 |
| API 接口已规划 | □ | 前后端分离时需明确接口规格 |
| 外部依赖已列出 | □ | 第三方库、服务、API |
| 技术栈已确定 | □ | 语言、框架、数据库等 |

### 4. 边界情况与异常处理 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 错误处理策略已确定 | □ | 如何处理异常、错误、失败 |
| 边界情况已考虑 | □ | 空值、极限输入、并发场景 |
| 日志记录已规划 | □ | 关键操作日志、错误追踪 |
| 降级方案已规划 | □ | 服务不可用时的处理 |

### 5. 配置与环境 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 环境配置需求已列出 | □ | 环境变量、配置文件 |
| 部署方案已考虑 | □ | 如何部署到生产环境 |
| 数据迁移需求已评估 | □ | 如有存量数据需考虑迁移 |
| CI/CD 流程已规划 | □ | 构建、测试、部署流程 |

### 6. 风险评估 ✅/❌/❓

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 技术风险已识别 | □ | 关键技术难点、未知领域 |
| 依赖风险已评估 | □ | 第三方服务稳定性、版本兼容 |
| 时间风险已考虑 | □ | 估算是否合理、buffer 是否足够 |

## 检查结果模板

```markdown
# 规划完整性检查报告

## 检查时间: [日期]
## 项目: [项目名称]

### 通过项目 ✅
- [ ] 功能需求完整
- [ ] 技术方案确定

### 缺失项目 ❌
- [ ] 性能要求未明确
- [ ] 错误处理策略未确定

### 需要明确项目 ❓
- [ ] 部署方案待定

### 建议补充信息
1. [具体建议1]
2. [具体建议2]
```

---

## 提示用户模板

当检测到缺失项目时，提示用户：

```
⚠️ 规划信息不完整

以下关键信息缺失，可能会导致计划不完整：

❌ 缺失项目：
  - 性能要求（响应时间、并发量）
  - 错误处理策略

❓ 需要明确：
  - 部署方案

建议补充这些信息后继续规划，以获得更完整的计划。
如需跳过检查继续规划，请输入 "跳过检查"。
```

---

## 使用说明

1. **规划前**：gsd-planner 自动检查此清单
2. **缺失提示**：提示用户补充关键信息
3. **可选跳过**：允许用户跳过继续规划
4. **记录状态**：在 PLAN.md 中记录检查状态
